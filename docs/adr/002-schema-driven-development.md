# ADR-002: OpenAPI を SSOT とした Schema-Driven Development の採用

## Status
* **Accepted**

## Context (背景と課題)
フロントエンドとバックエンドが分離されたモダンなWeb開発において、APIインターフェースの「仕様のズレ」は致命的な手戻り（Ping-Pong）を生み出す。
1. **型の不一致:** バックエンドがレスポンスのプロパティ名を変更（例: `userId` -> `user_id`）した際、フロントエンドがそれに気づかずデプロイされ、結合テスト（または本番）で画面がクラッシュする（Silent Failure）。
2. **モックの手動管理:** フロントエンド開発者が、バックエンドの実装を待たずに開発を進めるために手動でモックデータ（JSON）を作成している。仕様が変更された際、この手動モックが追従されず、テストが意味をなさない「嘘のテスト（The Liar）」になってしまう。

## Decision (決定事項)
APIの仕様書である **OpenAPI (Swagger) ファイルを SSOT (Single Source of Truth: 信頼できる唯一の情報源)** とし、そこから以下の2つをCI/CDパイプライン上で **「自動生成・同期」** させるアーキテクチャ（Schema-Driven Development）を採用する。

1. **フロントエンドの型定義 (`openapi-typescript` 等を使用)**
2. **MSW (Mock Service Worker) を用いた型安全なモックサーバー**

## Alternatives (代替案と不採用の理由)
* **BFF (GraphQL / tRPC) の導入:**
  * *理由:* GraphQLを導入すれば強力な型同期システムが手に入るが、既存のRESTfulなバックエンド資産をGraphQLに書き換えるコスト（学習コストとマイグレーションコスト）が高すぎるため、今回はRESTのまま自動化の恩恵を受けられる OpenAPI + TypeScript アプローチを採用。
* **TypeScriptの型定義を手動で共通リポジトリ（Submodule等）管理:**
  * *理由:* 手動での型定義は必ず「仕様書との乖離」を生む。ヒューマンエラーが介在する余地があるため不採用。

## Consequences (結果・影響)
### [Good] メリット
* **Drift（仕様と実装の乖離）の完全防止:** バックエンドがAPI仕様を変更した際、型生成コマンドを実行しない（あるいは型エラーを無視した）PRは、GitHub Actions のCIで自動的にブロックされる。これにより、インターフェースの不一致による本番障害がゼロになる。
* **開発ベロシティの向上:** フロントエンド開発者は、バックエンドの実装完了を待たずに、生成された型とMSWを用いて安全に先行開発（モック駆動開発）を進めることができる。

### [Bad] デメリット・トレードオフ
* **仕様書ファースト（Design-First）の強制:** バックエンド開発者は、コードを書く前に必ずOpenAPI仕様書を更新しなければならなくなる。
  * *Mitigation (緩和策):* FastAPI（Python）などのモダンなフレームワークを用いれば、コード（Pydanticモデル）からOpenAPI仕様書を自動生成（Code-First）できるため、この制約の負担を軽減できる。
